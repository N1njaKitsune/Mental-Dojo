Aura & Avatar

The visual progression system — 13 aura stages, avatar rendering, and integration points · v1.0 · March 2026

The Aura & Avatar system is not a section students navigate to — it's a visual layer that renders within screens across both the Nendō Dojo and Student Section. This spec covers what the system is, how it works, and where it surfaces. Build it once, integrate everywhere.

2. Avatar System

The avatar is a minimal, intentional visual representation that evolves with the student's aura progression. It consists of a silhouette body and head, with layered aura glows that intensify as the student progresses.

Avatar Structure

The avatar uses a simple capsule-shaped body silhouette with a circular head positioned above. The aura renders as concentric radial glows that expand outward. The avatar sits on a subtle ground plane that anchors the visual composition.

Avatar States

Dormant
No visible aura. Stage 1. Silhouette only.
Awakening
Faint glow appears. Stages 2–4. Single subtle layer.
Active
Clear aura visible. Stages 5–7. Multiple glow layers.
Radiant
Strong multi-layered aura. Stages 8–10. Particle effects.
Transcendent
Full visual presence. Stages 11–13. Complex layered rendering.

Rendering Approach

Avatar uses Canvas API for aura rendering. The body is CSS/SVG. Aura layers are concentric radial gradients with animated opacity. Particle effects use a lightweight particle system (see ParticleSystem component v1.0). This hybrid approach balances visual quality with performance on mobile devices.

3. Data Contract

The Aura system consumes and produces specific data structures. Understanding these inputs and outputs is critical for integration across screens and systems.

Inputs (What the system receives)

auraStage
integer (1–13)
Current aura stage number. Determines the visual representation and colour.
disciplineScores
object { metsuke, mushin, tachi: 0–100 }
Scores for the three NEN disciplines. Used to compute aura intensity and colour balance.
totalXP
integer
Cumulative experience points. Contributes to stage progression calculation.
streakDays
integer
Current training streak in days. Influences aura pulse animation intensity.
recentActivity
array (last 7 days)
Activity log for the last 7 days. Used to compute particle density and animation speed.

Outputs (What the system provides)

auraColour
string (rgba)
Computed RGBA colour for the current stage. Used for all visual rendering.
auraIntensity
float (0.0–1.0)
Glow strength computed from discipline scores and activity. Controls opacity and blur.
avatarLayers
array of layer objects
Layer definitions for Canvas rendering. Each layer specifies radius, colour, and opacity.
particleConfig
object (particle system config)
Configuration for the particle system: count, behaviour, colour, speed, lifetime.
stageName
string (Japanese)
Japanese name for the current stage (e.g., "Kagayaki" for stage 7).
phaseLabel
string
Phase label: "Foundation", "Growth", or "Mastery". Used for status displays.

Key architectural question: Does each screen import the Avatar component and render it, or does a global Avatar layer sit on top? Recommendation: each screen imports the component. The Avatar renders differently per context (full on Profile, orb-only on hubs, animation on session complete). A single global layer cannot handle these variations efficiently or cleanly. Distributed component ownership allows for flexible, context-aware rendering.

6. Development Approach

Two strategic approaches to building and integrating the Aura & Avatar system are presented below. Choose based on resource constraints and risk tolerance.

APPROACH A
Build Standalone,
Integrate Later
Build the Aura & Avatar as a self-contained component library. Test with dummy data using the existing AuraDisplay prototype as a harness. Once solid, integrate into each screen one at a time.
Pros
  • Clean separation of concerns
  • Testable in isolation
  • No dependencies on other screens' progress
  • Reusable across multiple contexts
Cons
  • Integration might reveal layout issues late
  • Extra step in the build pipeline
RECOMMENDED
APPROACH B
Build Into
Each Screen
Build the aura rendering directly into each screen as needed, evolving the approach with each integration. Start with the Profile, then hubs, then animations.
Pros
  • Immediate visual feedback in context
  • No separate integration step
  • Matches established component architecture
  • Faster time to first visual
Cons
  • Risk of inconsistency across screens
  • Potential code duplication
  • Harder to maintain across contexts

Recommendation: Approach B is recommended because it matches the component architecture already established with AuraOrb, ParticleSystem, and other reusable components. Each screen imports and composes the Avatar differently based on context, avoiding the overhead of a standalone build step while maintaining clean component boundaries.

8. Technical Requirements

The following technical rules govern the implementation and performance of the Aura & Avatar system.

Canvas API for aura rendering (not CSS shadows)
Must run at 60fps on iPad
Degrade gracefully on low-end phones
Particle count capped per device tier
Aura data cacheable — no recompute per frame
Avatar body is static SVG
Only aura animates
No external JS dependencies
Performance note: CSS shadows and filters are too slow on mobile for continuous animation. Canvas API provides better control and performance. Pre-compute aura layer definitions on data change, not on every frame. Cache computed colours and intensity values. Use requestAnimationFrame for smooth animations. Capped particle count prevents frame drops on older devices.
